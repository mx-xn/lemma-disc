## Coding standards — follow strictly

**Modularity and style**
- Write modular, principled code. Each function does one thing; keep helpers
  private (`_underscore`). Do not add abstractions beyond what the task
  requires. Three similar lines are better than a premature abstraction.
- Comments only when the *why* is non-obvious (a hidden constraint, a subtle
  invariant, a known upstream quirk). Never describe what the code does — the
  names do that. No multi-line comment blocks.
- Type hints on every function signature. Dataclasses for structured data;
  no raw dicts past the JSON parsing boundary.

**Testing**
For each module, decide whether a test is warranted and act accordingly:
- If yes: write a short, focused test (a few cases, no elaborate fixtures).
  Place it as `test_<module>.py` alongside the module.
- If no: add a one-sentence comment at the top of the module explaining why
  (e.g. "thin dispatch — correctness verified by the delegated stage's own
  tests").
Tests should cover the real logic, not trivial boilerplate. Keep them short —
do not bloat the codebase.

**Runtime environment**
All commands are run inside the **`lemma` conda environment**. When invoking
subprocesses (e.g. `pipeline/run_full.sh`, `lake env lean`), prefix with the
conda env's bin path or assume the env is already activated. Do not hardcode
absolute paths to binaries; use `shutil.which` or rely on the activated env's
`PATH`.

## What this is

A one-shot train/test evaluation: given N theorems from a single domain (a
single Lean source file with complete proofs), split them into a training set X
and a test set Y; learn a lemma library L from X; evaluate LLM proof automation
on Y under three conditions:

- **baseline**: no lemmas (vanilla LLM)
- **with-L (pipeline)**: lemmas extracted algorithmically via phases 1–5
- **with-L (llm)**: lemmas generated by an LLM reading the training corpus

The key distinction from `exp/eval/`'s iterative loop: learning happens on
**complete proofs from the training set**, not from partial proofs of the test
set. There is no iteration — one split, one learn, one eval.

## Input formats

### `domain_theorems.json`
Same schema as `exp/expB/input/theorems.json` and `exp/eval/sanity/output/theorems_sanity.json`:
```json
{
  "lake_project": "/path/to/lean/project",
  "imports": ["LeanSrc.Domain"],
  "theorems": [
    {
      "theorem_id": "thm_01",
      "lean_path": "LeanSrc/Domain.lean",
      "statement_text": "(n : Nat) : n + 0 = n",
      "local_ctx": ""
    }
  ]
}
```
`local_ctx` must include any `open`/`namespace` blocks or local definitions
from the source file that the theorem depends on. Same authoring responsibility
as expB.

### Digestion trace (`--trace`)
Output of `python -m digestion.src.extractor --local <project>`.
One JSON file per source file; schema: `schemas/trace.schema.json`.
The `learn --method pipeline` command filters this to training declarations by
matching `declarations[].name` against theorem IDs from `train.json`.

## Commands

### `split`
```
python -m exp.eval.domain_eval.run split
    --theorems  PATH   (default: input/domain_theorems.json)
    --output-dir PATH  (default: output/)
    --train-frac FLOAT (default: 0.7; mutually exclusive with --train-n)
    --train-n    INT
    --seed       INT   (default: 42)
    --force
```
Writes `<output-dir>/train.json` and `<output-dir>/test.json`.
Both files are valid TheoremSets (preserve `lake_project` and `imports`).

### `learn`
```
python -m exp.eval.domain_eval.run learn
    --method     {pipeline, llm}  (default: pipeline)
    --train      PATH             train.json from split step
    --output-dir PATH             (default: output/)
    --lake-project PATH           (required for lemma_fixer; overrides train.json)
    --imports    STR              comma-separated (overrides train.json)
    --skip-fix                    skip lemma_fixer (for testing)
    --force

    # pipeline only:
    --trace      PATH             trace.json from digestion
    --skip-pipeline               skip run_full.sh; write empty raw_lemmas.json
                                  (use while Scala phases 2-4 are incomplete)

    # llm only:
    --lean-file  PATH             Lean source with training proofs only
    --model      STR              (default: gpt-5.4-mini)
```
Writes:
- `train_trace.json` (pipeline only) — filtered trace
- `raw_lemmas.json` — statements before fixing (`{"lemmas": [...]}`)
- `learned_lemmas.json` — validated, fixed statements (same format)

**Data-isolation note**: `--lean-file` must contain only training material.
If the domain file contains both train and test proofs, the user must produce
a training-only copy before passing it here.

### `validate`
```
python -m exp.eval.domain_eval.run validate
    --lemmas     PATH   raw_lemmas.json from learn step (default: output/raw_lemmas.json)
    --train      PATH   train.json to read lake_project/imports (default: output/train.json)
    --output-dir PATH   (default: output/)
    --lake-project PATH overrides train.json
    --imports    STR    comma-separated (overrides train.json)
    --force
```
Interactive step between `learn` and `eval`. Pre-filters obviously malformed
statements (truncated, unbalanced brackets), Lean-checks the rest in one batch,
then prompts `[d]iscard / [e]dit` for each failure. Writes `learned_lemmas.json`.
Use `--skip-fix` on `learn` to skip the automated LLM fixer and rely on this
step instead.

### `eval`
```
python -m exp.eval.domain_eval.run eval
    --theorems   PATH   test.json from split step
    --output-dir PATH   (default: output/eval/)
    --lemmas     PATH   learned_lemmas.json; if absent → baseline (no hints)
    --top-k      INT    retrieve top-K lemmas per theorem via ByT5Retriever
                        (only meaningful with --lemmas; omit to use full pool)
    --model      STR    (default: gpt-5.4-mini)
    --max-attempts INT  (default: 3)
    --workers    INT    (default: 1)
    --lake-project PATH (overrides test.json)
    --imports    STR    (overrides test.json)
    --force
```
Writes `prove_results.json` — same schema as `exp/eval/stages/prove.py` output.

## Implementation notes

### `learn --method pipeline` internals
1. Load `trace.json` (trace.schema.json). Keep only `declarations` whose
   `name` ∈ training `theorem_id` set. Write `train_trace.json`.
2. Subprocess: `bash pipeline/run_full.sh <train_trace.json> <raw_lemmas.json>`
   (skip if `--skip-pipeline`; write `{"lemmas": []}` instead).
3. Parse `raw_lemmas.json` (lemmas.schema.json); collect `lemmas[].statement`.
4. Run `fix_lemmas` from `exp.eval.lib.lemma_fixer` (skip if `--skip-fix`).
5. Write `learned_lemmas.json`.

### `learn --method llm` internals
1. Read Lean source text from `--lean-file`.
2. Render `exp/eval/prompts/llm_generate.md` with `<<corpus>>` = file text,
   `<<imports>>` = comma-joined imports list.
3. Call LLM; split response on newlines; strip blank lines → raw statements.
4. Write `raw_lemmas.json` (`{"lemmas": [...]}`).
5. Run `fix_lemmas` (skip if `--skip-fix`).
6. Write `learned_lemmas.json`.

### `eval` internals
This stage contains no proving logic of its own — it dispatches:

- `--lemmas` absent → call `run_baseline(theorems_path=test.json, output_dir=...,
  ...)` from `exp.eval.stages.baseline`. Output: `01_baseline.json`.
- `--lemmas` present → load pool from `learned_lemmas.json`; instantiate
  `ByT5Retriever()` if `--top-k` given (else `retriever=None`); call
  `prove_round(theorems, lemma_pool, ..., retriever=retriever, top_k=top_k)`
  from `exp.eval.stages.prove`. Output: `prove_results.json`.

The `prove_round` API already accepts `retriever` and `top_k`; no changes to
that module are needed.
